# 三大核心原理

## 原理一：AI 是目标开发者

**定义**：项目的文档、结构、流程都为 AI 设计，人类可读性是次要目标。

**推论**：
- 目录结构要考虑 AI 的上下文窗口限制
- 文档要用结构化格式（表格、列表）而非长段落
- 状态信息要显式标注，不能依赖"常识"

**反例**：
```markdown
# 差：人类友好但 AI 难以解析
这个项目主要是做用户管理的，有一些 API 接口，
代码在 target 目录下，有些已经完成了，有些还在开发中...
```

**正例**：
```markdown
# 好：结构化，AI 可快速理解
| 功能 | 状态 | 代码位置 |
|------|------|---------|
| 用户登录 | ✅ 完成 | target/auth/login.ts |
| 用户注册 | 🚧 进行中 | target/auth/register.ts |
```

---

## 原理二：有效上下文

**定义**：AI 的输出质量取决于输入的上下文质量。上下文过长会导致"上下文腐化"，过短会导致信息不足。

**推论**：
- 使用分层索引，按需加载
- 表格化信息，减少冗余
- 索引文件只存路由，不存内容

**上下文控制策略**：

```
Layer 0: CLAUDE.md      ~1.5K tokens  ← 始终加载
Layer 1: workspace/     ~2K tokens    ← 按项目加载
Layer 2: 具体文档       按需          ← 只读必要的
```

---

## 原理三：规范驱动开发 (SDD)

**定义**：先定义规范，再由 AI 执行。规范 > 代码。

**推论**：
- 没有规范不写代码
- 代码必须符合规范
- 规范变更需要显式记录

**SDD 工作流**：

```
1. 定义规范 (convention/)
2. AI 阅读规范
3. AI 生成代码 (target/)
4. 验证代码符合规范
5. 归档
```

**与传统 TDD 对比**：

| 维度 | TDD | SDD |
|------|-----|-----|
| 驱动力 | 测试用例 | 规范文档 |
| 适用范围 | 代码层面 | 全流程 |
| 执行者 | 人类 | AI |
