# 与传统开发对比

## 对比表

| 维度 | 传统开发 | AI Native |
|------|----------|-----------|
| **代码生产者** | 人类 | AI |
| **文档角色** | 代码的副产品，往往过时 | 核心交付物，持续更新 |
| **知识存储** | 隐式（人脑、隐性知识） | 显式（结构化文档） |
| **新成员上手** | 需要导师带领，周期长 | 读 CLAUDE.md 即可开始 |
| **工作流** | 因人而异 | 标准化阶段结构 |
| **可追溯性** | 依赖提交信息 | 从需求到部署全程记录 |
| **质量控制** | 代码评审 | 规范验证 + 代码评审 |

## 思维转变

### 从"怎么写"到"写什么"

传统开发关注"怎么实现这个功能"（算法、设计模式）。
AI Native 关注"这个功能应该是什么"（规范、验收标准）。

### 从"代码为中心"到"文档为中心"

```
传统:   需求文档(可选) → 代码 → 测试 → 上线
AI Native: 规范文档(必需) → AI生成代码 → 验证 → 上线
```

### 从"记忆"到"索引"

传统开发依赖开发者记住项目细节。
AI Native 依赖结构化索引，AI 随时可以查阅。

## 适用场景

**适合 AI Native 的场景**：
- 重复性高的业务逻辑
- 有清晰规范可定义的领域
- 团队成员变动频繁
- 需要快速迭代

**暂不适合的场景**：
- 探索性研发，规范不明确
- 极度复杂的算法优化
- 需要深度领域专家经验的工作
