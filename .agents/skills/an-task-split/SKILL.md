# AI Native 任务拆分

将大型任务拆成可独立推进、可验证、可排期的子任务。

## 输入来源

- 用户直接描述的大任务。
- `workspace/**/raw-input/task.md`。
- 产品文档、issue、设计稿或路线图链接。

## 流程

```text
原始输入 -> 拆分方案 -> 可选创建工作区
```

## 阶段一：原始输入

创建总任务工作区：

```text
workspace/{YYYYMMDD}__{feature-name}/raw-input/task.md
```

原样保存用户输入和引用来源。

## 阶段二：split-plan

从业务场景、用户路径、技术边界、依赖关系和验证方式拆分。

产出：

```markdown
# {任务名称} 拆分方案

## 任务理解

## 拆分依据

## 拆分概览
| 序号 | 子任务 | 目标 | 依赖 | 建议流程 |
|------|--------|------|------|----------|

## 依赖关系

## 子任务定义

### 1. {sub-task}
- 目标:
- 关键验收标准:
- 涉及工程:
- 建议工作区:
- 风险:

## 执行顺序建议
```

## 确认后动作

用户确认拆分方案后：

1. 更新总任务工作区索引。
2. 为用户选择的子任务创建独立工作区，或只记录建议工作区路径。
3. 更新根 `AGENTS.md` 的活跃工作区表。

## 拆分原则

- 每个子任务应能独立验收。
- 子任务边界优先跟随业务闭环，其次跟随技术模块。
- 避免把同一文件或同一接口拆到多个并行子任务造成冲突。
- 如果任务并不大，说明无需拆分并建议使用 `$an-task`。
