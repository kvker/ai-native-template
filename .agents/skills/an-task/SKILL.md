# AI Native 任务开发

将用户输入转化为可追踪的工程任务，并推进到实现和验证。

## 输入来源

- 用户直接输入的描述。
- 本地文件、issue、设计稿或外部文档链接。
- 已有 `workspace/` 中的历史任务。

如果用户提到“上次”“之前”“继续某个任务”等历史意图，先只匹配 `workspace/` 和 `workspace/archive/` 的目录名；用户确认后再读取内容。

## 判断流程范围

| 信号 | 范围 | 行为 |
|------|------|------|
| 拼写、注释、明显一行 bug、无行为风险的小配置 | L0 直接修复 | 不强制创建工作区，直接改并验证 |
| 单模块、小范围、行为明确、有现成验证 | L1 快速修复 | 保存原始输入，直接实现，补充实现和测试记录 |
| 用户已有明确方案或接口约定 | L2 标准开发 | 保存原始输入，从技术规范开始，再实现、验证、质量评价 |
| 需求模糊、新功能、跨模块、架构决策 | L3 完整流程 | 原始输入 -> 需求 -> 设计 -> 技术规范 -> 实现 -> 测试 -> 发布归档 -> 质量评价 |

判断依据见 `.agents/rules/flow-policy.md`。判断后先告诉用户将采用哪个等级。L0/L1 可直接完成后总结；L2 默认技术规范后确认；L3 需要关键阶段确认。

## 升级条件

出现以下情况时，至少升级到 L2，高风险时升级到 L3：

- 公开接口、数据库结构、认证鉴权、支付、权限、部署配置。
- 跨工程或跨核心模块。
- 需求存在开放问题或验收标准不明确。
- 需要新增依赖、数据迁移、外部服务或破坏性命令。
- 测试缺失且改动影响用户可见行为。

## 工作区创建

使用当天日期和简短英文特性名：

```text
workspace/{YYYYMMDD}__{feature-name}/
├── AGENTS.md
├── raw-input/
├── requirements/
├── design/
├── tech-spec/
├── implementation/
├── testing/
└── deployment/
```

工作区 `AGENTS.md` 作为任务索引：

```markdown
# {任务名称}

## 状态

| 阶段 | 文档 | 状态 |
|------|------|------|
| raw-input | [task](raw-input/task.md) | 完成 |

## 关键决策

| 日期 | 决策 | 原因 |
|------|------|------|
```

同步更新根 `AGENTS.md` 的活跃工作区表。

## 阶段产出

### 原始输入

保存原始输入，不做加工：

```text
raw-input/task.md
```

### 需求

```markdown
# {任务名称} 需求文档

## 背景与目标

## 验收标准
- [ ] 验收标准 1：{可验证标准}

## 范围
### 包含
### 不包含

## 依赖

## 涉及模块

## 开放问题
```

### 设计

```markdown
# {任务名称} 技术方案

## 方案概述

## 决策表
| 决策点 | 选项 | 选择 | 理由 |
|--------|------|------|------|

## 风险与权衡
```

### 技术规范

```markdown
# {任务名称} 技术规范

## 数据模型或接口

## 核心逻辑

## 变更清单
| 序号 | 文件 | 变更类型 | 说明 |
|------|------|----------|------|

## 测试计划
```

### 实现

按技术规范实现。遇到规范未覆盖的问题时：

1. 先选择最小、安全、符合现有代码风格的方案。
2. 在 `implementation/decisions.md` 记录原因。
3. 如果影响需求或架构边界，回到对应阶段更新文档并确认。

### 测试

优先读取 `.agents/recipes.json` 选择验证命令；不存在时运行 `$an-recipes` 探测。记录：

```markdown
# 测试报告

## 执行记录
| 命令 | 结果 | 说明 |
|------|------|------|

## 缺陷记录
| 描述 | 严重程度 | 状态 |
|------|----------|------|
```

### 质量评价

L2/L3 在测试后运行：

```bash
node .agents/skills/an-eval/scripts/evaluate-task.mjs workspace/{YYYYMMDD}__{feature-name}
```

保存为 `testing/eval-report.md`。如果结论为 `BLOCKED`，不要归档任务；先补验收标准、测试证据或风险处理。

### 发布归档

需要发布或收尾时生成：

```markdown
# 发布说明

## 变更摘要

## 上线清单

## 回滚方案
```

## 调查和实现约束

- 先读根 `AGENTS.md`、相关目录 `AGENTS.md`、必要的 `background/` 文档。
- 用 `rg` / `rg --files` 精确定位代码。
- 不读取密钥、依赖、构建产物和大型生成文件。
- 不自动使用子代理；只有用户明确要求代理协作时才委派。
- 修改代码后优先运行 `.agents/recipes.json` 中最小相关命令；没有命令清单时运行项目中最接近的测试、类型检查、代码检查或构建命令。
- 无法运行验证时，说明原因和剩余风险。

## 完成输出

向用户说明：

- 采用的流程范围。
- 修改的主要文件。
- 运行的验证命令和结果。
- L2/L3 的质量评价结论和主要扣分项。
- 需要用户确认或后续处理的事项。
