## 核心原则

1. AI 是目标开发者：文档为 Codex 的执行效率设计，人类可读是次要目标。
2. 有效上下文优先：入口只放路由和稳定约定，细节在需要时再读取。
3. 规范驱动开发：重要改动先形成需求、设计或技术规范，再实现和验证。
4. 代码事实优先：`target/` 中的实际代码和测试结果优先于过时文档。
5. 人类决策显式化：需求范围、架构取舍、跳过流程和高风险操作都要记录。

## Codex 工作约定

- 搜索文件和文本优先使用 `rg` / `rg --files`。
- 读取上下文时先看路由文件，再读取相关小片段；避免一次性加载整个 `target/`。
- 不读取 `.env*`、`secrets/**`、凭据文件、构建产物、依赖目录和大型生成文件，除非用户明确要求。
- 修改代码时保持范围小，尊重已有未提交变更，不回滚用户改动。
- 涉及第三方接口、框架版本或外部规则时，优先使用官方文档或本地锁定版本，不凭空补全。
- 前端或交互改动需要验证时，按项目已有方式启动本地服务并说明访问地址和验证结果。
- 只有用户明确要求使用子代理、并行代理或委派时，才启动 Codex 子代理；普通任务用本地搜索和阅读推进。

## 技能路由

| 意图 | 推荐技能 |
|------|----------|
| 将已有项目迁入 AI Native 结构 | `$an-init` |
| 开始一个功能、修复、重构或文档任务 | `$an-task` |
| 拆分大型需求或长任务 | `$an-task-split` |
| 探测或刷新测试、构建、代码检查、生成命令 | `$an-recipes` |
| 根据代码和已完成任务刷新背景知识 | `$an-refresh` |
| 评价任务是否真的完成 | `$an-eval` |
| 归档已完成的工作区 | `$an-archive` |

技能被调用时，先读取对应的 `SKILL.md`，再按其中流程执行。未显式调用技能但用户意图明显匹配时，可以主动说明将使用哪个 AI Native 流程。
